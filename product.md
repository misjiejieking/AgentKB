# 二、产品概述

## 2.1 产品背景

当前个人知识管理存在以下问题：

- 笔记分散在 Markdown、文档、聊天记录中
- 文件搜索依赖关键词，无法自然语言理解
- 企业制度、学习资料难以统一查询
- AI工具依赖云端，存在隐私与数据安全问题
- 本地部署门槛高，配置复杂

随着本地大模型与 Agent 技术成熟，用户需要一个：

> “能够在本地运行、理解个人知识、支持工具调用的智能助手。”

因此，本产品将基于 LangGraph 构建一个：

- 本地优先
- 轻量部署
- 支持知识检索
- 支持工具调用
- 可扩展 Agent 能力

的个人知识助手。

---

# 三、产品定位

## 3.1 产品定位

一款基于 LangGraph 的本地优先 AI Agent 系统，帮助用户统一管理个人知识、企业文档与联网信息，并通过自然语言实现高效问答与任务协作。

---

## 3.2 产品愿景

打造一个：

> “真正属于用户自己的本地 AI Agent 操作系统”

让用户拥有：

- 私有知识大脑
- 本地智能助手
- 可扩展工具生态
- 长期记忆能力

# 九、技术架构设计

## 9.1 总体架构

```text
前端层（Gradio）
    ↓
LangGraph编排层
    ↓
能力层
  ├── LLM服务
  ├── 工具系统
  ├── 知识库系统
  └── 会话系统
    ↓
本地基础设施
  ├── SQLite
  ├── Qdrant
  ├── FileSystem
  └── Memory Cache
```

---

## 9.2 技术选型

| 模块 | 技术 |
|---|---|
| Agent编排 | LangGraph |
| LLM框架 | LangChain |
| UI | Gradio |
| 向量库 | Qdrant |
| 数据库 | SQLite |
| Embedding | BGE-M3 |
| 搜索 | DuckDuckGo |
| 本地模型 | Ollama |

---

## 9.3 架构设计原则

### 原则一：本地优先

所有核心能力支持离线。

---

### 原则二：最小依赖

不依赖：

- Docker
- Redis
- PostgreSQL
- Kubernetes

---

### 原则三：模块解耦

模块之间通过接口通信：

- 工具模块
- LLM模块
- 知识库模块

可独立替换。

# 十、LangGraph工作流设计

## 10.1 Agent流程

```text
用户输入
    ↓
消息预处理
    ↓
LLM决策
    ↓
是否调用工具
  ├── 否 → 直接回复
  └── 是
        ↓
    工具执行
        ↓
    结果整合
        ↓
    最终生成
```

---

## 10.2 Agent状态定义

```python
class AgentState(TypedDict):
    messages: Sequence[BaseMessage]
    session_id: str
    retrieved_docs: list[dict]
    tool_calls: list[dict]
    tool_results: list[dict]
```
