# AgentKB 项目深度优化分析报告

> 审查日期：2026-05-23
> 审查范围：全部核心模块（graph / nodes / retriever / reranker / chunker / pg_database / evaluator / tools / ui / api）
> 视角：系统架构师 + 产品经理 双重视角

---

## 一、系统核心能力提升

### 1.1 检索质量

#### 1.1.1 【高优先级】查询重写（Query Rewriting）

**现状**：用户输入直接传给 `retriever.retrieve(query)`，无任何预处理。如果用户问 "上次说的那个制度怎么规定的"，零指代消解和上下文展开完全缺失。

**建议**：在检索前增加 LLM 查询重写节点，执行三项操作：
- 指代消解：结合历史对话将 "它"/"那个" 展开为具体实体
- 多路查询：将复杂问题拆为 2～3 个子查询分别检索后合并
- 关键词提取：生成 BM25 友好的关键词版本与语义版本并行检索

**预期收益**：Recall@5 预计可提升 15～30%（尤其对多轮对话场景）
**实施难度**：中（需在图中增加一个 pre-retrieval 节点，约 80 行代码）

---

#### 1.1.2 【高优先级】检索幻觉检测 + 拒答机制

**现状**：`knowledge_search.py` 即使在 `candidates` 为空时也返回 `success=True`。LLM 在没有相关文档时仍会尝试回答，可能导致编造。System prompt 虽然要求 "诚实告知"，但本地小模型（qwen2.5:7b）遵从性差。

**建议**：
- 检索结果返回 confidence score，低于阈值时明确标记 `relevant: false`
- System prompt 增加强约束：当所有检索结果的 relevance 标记为 false 时，仅回复 "知识库中未找到相关信息" 并停止
- 增加 source citation 的 post-hoc 验证

**预期收益**：幻觉率显著降低，用户信任度提升
**实施难度**：低（主要改 prompt + `knowledge_search._execute` 增加阈值判断）

---

#### 1.1.3 【中长期】HyDE（Hypothetical Document Embeddings）

**现状**：embedding 模型直接编码用户 query，但 query 和文档在向量空间中天然有 gap（query 是问题，文档是陈述）。

**建议**：检索前先用 LLM 生成一份假设答案（5～10 个 tokens 的陈述句），再用假设答案的 embedding 去检索。

**预期收益**：Recall@5 典型提升 5～15%，尤其是对简短/模糊 query
**实施难度**：中（额外一次 LLM 调用，增加约 300ms 延迟）

---

#### 1.1.4 【中长期】迭代式检索（Iterative Retrieval）

**现状**：每次检索是 one-shot 的。Agent 调用一次 `search_knowledge_base` 后就直接生成答案，不会判断检索质量并尝试改进。

**建议**：让 Agent 在收到检索结果后评估质量，必要时重新检索（换关键词、放宽 scope）。LangGraph 的 ReAct 循环天然支持，只需改进系统 prompt 使 Agent 能自我判断检索是否充分。

**预期收益**：复杂问题的答案完整度提升 20%+
**实施难度**：中（主要改 prompt 逻辑，需要好的判断标准）

---

### 1.2 生成准确性

#### 1.2.1 【高优先级】上下文窗口管理

**现状**：`knowledge_search.py` 返回的检索结果全部塞给 LLM，没有考虑上下文窗口限制。当 `parent_content` 较长（1024 tokens）且返回 5 条时，仅检索内容就占 5000+ tokens。

**建议**：
- 计算每条检索结果的 token 数，根据模型 max_tokens 动态截断
- 优先保留 rerank_score 高的内容
- 对 parent_content 做 smart truncation（保留包含关键句的窗口而非全文）

**预期收益**：避免上下文溢出导致的生成截断/遗忘，大文件场景尤其重要
**实施难度**：低（在 `knowledge_search._execute` 末尾加约 30 行截断逻辑）

---

#### 1.2.2 【中长期】生成质量评估（RAGAS-style）

**现状**：评估模块（`eval/`）只覆盖检索指标（Recall/MRR/NDCG），没有生成质量指标（Faithfulness/Answer Relevance/Context Relevance）。

**建议**：引入轻量级生成评估：
- Faithfulness（忠实度）：LLM 判断答案是否完全来自检索上下文
- Answer Relevance：LLM 判断答案是否回答了问题
- 每次评估运行后自动统计幻觉率

**预期收益**：从 "检索好不好" 升级到 "答案对不对"，形成真正的质量闭环
**实施难度**：高（需要 LLM-as-judge 的 prompt 工程 + 大量测试调优）

---

### 1.3 Agent 协作

#### 1.3.1 【高优先级】工具选择策略优化

**现状**：`prompts.py` 中工具选择完全靠 prompt 指令（"优先使用 search_knowledge_base"），小模型容易出现工具选择错误。

**建议**：
- 增加工具选择的 few-shot examples 到 system prompt
- 考虑实现简单的意图分类 router：识别到问候/闲聊时跳过所有工具直接回复
- 工具描述增加负面示例（"不要用这个工具做什么"）

**预期收益**：工具选择准确率提升，减少不必要的检索调用（降低延迟）
**实施难度**：低（主要改 prompt，约 20 行）

---

#### 1.3.2 【中长期】Multi-Agent 编排

**现状**：单 Agent 模型。检索和生成由同一个 LLM 完成，无法针对不同任务使用不同模型。

**建议**：
- Router Agent：负责意图识别和任务分发
- Retriever Agent：专门负责知识检索（可用更小的模型）
- Generator Agent：负责答案生成（可用更强的模型或云端模型）
- 异步并行：Router 判断需要检索时，检索和 web search 并行执行

**预期收益**：延迟降低 30～50%（小模型做路由，大模型只做生成），各环节可独立优化
**实施难度**：高（需要重构 LangGraph 图结构，引入多模型配置）

## 二、产品体验优化

### 2.1 响应感知

#### 2.1.1 【高优先级】首 token 延迟优化

**现状**：从用户提问到看到第一个 token，当前链路是 `query → embed → dense search → bm25 → RRF → rerank → LLM`。Embedding 模型在 CPU 上耗时可能达到 200～500ms。

**建议**：
- Web search 和 knowledge search 在 Agent 做工具决策时并行预热（speculative tool execution）
- 或更简单：将 embedding 模型切换为 ONNX Runtime 加速推理（通常提速 2～4x）

**预期收益**：首 token 延迟降低 200～400ms
**实施难度**：中（ONNX 转换 + 集成）

---

#### 2.1.2 【中长期】流式检索状态推送

**现状**：`graph.py` 的 `stream()` 通过 `on_tool_start`/`on_tool_end` 推送工具状态，但工具内部（检索 → 重排）是黑盒。

**建议**：在检索工具内部增加进度推送（"正在向量检索…" → "找到 30 条候选，正在精排…" → "已返回 5 条最相关结果"）。

**预期收益**：用户感知延迟降低（等待时知道在做什么）
**实施难度**：中（需要工具支持回调或 yield 中间状态）

---

### 2.2 交互方式

#### 2.2.1 【高优先级】Gradio UI 现代化

**现状**：`ui/app.py` 的 Gradio 界面非常基础（单一聊天框 + 上传按钮）。对话体验用的是 `gr.Chatbot` 基础组件。

**建议**：
- 增加侧边栏会话列表（多会话切换，已有 API 支持）
- 文件管理面板（已上传文件列表 + 删除按钮，已有 API 支持）
- 引用高亮：答案中的 `[来源: xxx]` 渲染为可点击链接
- 支持 Markdown 表格/代码块的 syntax highlighting

**预期收益**：从 "能用的 demo" 升级到 "可日常使用的产品"
**实施难度**：中（前端改造，约 200 行代码）

---

#### 2.2.2 【中长期】Web UI 替代 Gradio

**现状**：Gradio 适合原型但不利于产品化——定制能力弱、移动端体验差、无法做细粒度 UI 控制。

**建议**：评估是否迁移到：
- Next.js + FastAPI（最大灵活性，独立前端项目）
- 或 Streamlit（比 Gradio 更现代但同样是 Python UI）
- 或保留 Gradio + 自定义 JS/CSS 深度定制

**预期收益**：企业级用户体验，支持移动端
**实施难度**：高（需独立前端工程）

---

### 2.3 透明度

#### 2.3.1 【高优先级】答案溯源 + 可信度标识

**现状**：`knowledge_search.py` 返回结果中已有 `filename` 和 `score`，但前端未充分展示。用户看不到每条引用来自哪个文件、相关度多高。

**建议**：
- 前端渲染答案时，在引用处添加 tooltip（来源文件名 + 相关性分数）
- 答案结尾增加 "引用来源" 卡片
- 对低分检索结果，答案开头标注 "⚠️ 以下内容参考了相关性较低的文档片段"

**预期收益**：用户信任度和可验证性大幅提升
**实施难度**：低（前端展示逻辑，约 50 行代码）

---

### 2.4 失败处理

#### 2.4.1 【高优先级】优雅降级链路

**现状**：`web_search.py` 有两层回退（库 → HTML 解析），但 knowledge search 没有任何降级——dense search 失败即整体失败。

**建议**：
- 知识检索三层降级：混合检索 → dense-only → BM25-only → 空结果提示
- LLM 调用失败时：自动重试 1 次 → 返回 "服务暂时不可用" 而非技术错误栈
- 任一环节超时不阻塞整体（当前 `request_timeout: 120` 但未在检索链路中生效）

**预期收益**：鲁棒性从 "单点故障" 升级到 "尽力服务"
**实施难度**：中（在各模块增加 try/fallback 逻辑）

## 三、工程稳定性与规模化

### 3.1 延迟优化

#### 3.1.1 【高优先级】Embedding 推理 GPU 支持

**现状**：`config.yaml` 中 `embedding.device: cpu`。BGE-M3 在 CPU 上编码单条 query 需约 200ms，批量编码文档更慢。`embedder.py` 中 `local_files_only=True` 强制本地模型，但没用 ONNX/量化。

**建议**：
- 支持 CUDA/MPS 设备选择，启动时自动检测最优设备
- 使用 `sentence-transformers` 的 `model.half()` 做 FP16 推理（提速 2x，精度损失可忽略）
- 或导出 ONNX 模型用 ONNX Runtime 推理

**预期收益**：检索延迟降低 50～70%
**实施难度**：低（改配置 + `embedder.py` 增加设备检测，约 20 行）

---

#### 3.1.2 【中长期】检索结果缓存

**现状**：每次对话独立执行完整检索链路，高频问题的结果重复计算。同一 session 内相同 query 会重复检索。

**建议**：
- 语义缓存：对 query embedding 做 LSH/L2 近似匹配，命中则直接返回缓存结果
- TTL 策略：知识库更新后失效相关缓存
- 简单的 LRU 内存缓存即可满足 MVP

**预期收益**：高频问题响应延迟降至约 50ms（缓存命中时）
**实施难度**：中（需设计缓存 key / 失效策略）

---

### 3.2 成本优化

#### 3.2.1 【中长期】多模型分层策略

**现状**：所有环节（Agent 决策 + 答案生成 + 评估中的 LLM-as-judge）共用同一个 LLM（默认为 qwen2.5:7b）。

**建议**：
- Agent 决策（tool selection）→ 小模型（如 qwen2.5:3b），延迟低
- 答案生成 → 大模型（qwen2.5:14b 或云端 API）
- 评估 judge → 中等模型
- 查询重写 → 同上小模型

**预期收益**：在不牺牲答案质量的前提下，Agent 决策延迟降低 40%，资源占用降低
**实施难度**：高（需要多模型配置 + 路由逻辑）

---

### 3.3 可观测性

#### 3.3.1 【高优先级】全链路 Trace

**现状**：日志仅靠 loguru 输出到文件，没有结构化 trace。出现问题后难以定位是检索差、重排差还是 LLM 幻觉。

**建议**：
- 使用 LangSmith / LangFuse / 自建 trace 系统记录每次 Agent 调用的完整链路
- 每个 `tool_call` 记录：query → candidates → rerank_scores → final_results
- 每个 LLM 调用记录：input tokens / output tokens / 延迟
- 前端展示 "本次查询耗时分解"

**预期收益**：问题定位从小时级降至分钟级，迭代效率质变
**实施难度**：中（LangFuse 集成约 100 行代码，免费套餐够用）

---

#### 3.3.2 【中长期】用户行为埋点

**现状**：无任何用户行为追踪。

**建议**：
- 匿名埋点记录：问答频率、检索空结果率、用户复制答案/点赞率
- Dashboard：最常问的问题、知识库覆盖率、用户满意度趋势

**预期收益**：从 "猜测用户需求" 升级到 "数据驱动产品迭代"
**实施难度**：中（需要前端打点 + 后端统计 API）

---

### 3.4 迭代闭环

#### 3.4.1 【高优先级】用户反馈回路

**现状**：无任何用户反馈机制。用户无法标记 "答案有用"/"答案不对"。

**建议**：
- 每个答案下方添加 👍/👎 按钮
- 👎 时弹出简短的反馈理由（"不相关"/"不准确"/"不完整"）
- 反馈数据存入 DB，作为评估测试集的补充来源
- 定期自动从低分答案中抽样生成新的评估用例

**预期收益**：测试集从 "人工标注" 扩展到 "用户反馈驱动"，覆盖真实问题分布
**实施难度**：低（API 已有，需前端按钮 + 一个反馈表 + 约 80 行代码）

---

#### 3.4.2 【中长期】Continuous Eval（CI/CD 集成）

**现状**：评估模块（`eval/cli.py`）是手动运行的 CLI 工具。

**建议**：
- GitHub Actions workflow：每次 PR 触发时自动运行评估套件
- 对比基线：自动生成 DiffReport，Recall 退化超过 2% 时 PR 被 block
- 定期（每日）从生产数据采样生成新测试集

**预期收益**：防止检索质量无声退化，形成 "改代码 → 跑评估 → 看结果" 的工程闭环
**实施难度**：中（CI 配置 + 测试数据管理）

## 四、商业价值与用户价值提升点

### 4.1 【高优先级】构建 "知识库健康度" 仪表板

**现状**：用户上传文件后对索引质量无感知——不知道哪些文件 chunk 效果差、哪些内容没有被覆盖到。

**建议**：
- 对已上传文件自动运行评估，给每个文件打分（平均 Recall@5）
- 低分文件标记为 "建议优化"（可能是格式问题、内容太碎等）
- 提供 "知识库覆盖度" 总览：你的知识库能回答哪些类型的问题

**预期收益**：从 "上传即遗忘" 到 "主动管理知识资产"，提升用户粘性
**实施难度**：中（复用现有 eval 模块 + 前端仪表板）

---

### 4.2 【中长期】企业知识空间（多租户）

**现状**：单一知识库，所有用户共享。PRD 中 P2 已规划但未实现。

**建议**：
- 增加 `workspace` 概念：每个 workspace 有独立的知识库、会话和配置
- 支持 workspace 级别的权限（私有/共享/公开）
- 这是企业商业化的关键功能

**预期收益**：打开 B 端市场，支撑商业定价
**实施难度**：高（DB schema 变更 + 权限模型 + 前端改造）

---

### 4.3 【中长期】插件/MCP 工具生态

**现状**：PRD 已规划 MCP 支持但未实现。当前只有 2 个硬编码工具。

**建议**：
- 实现 MCP Client，支持用户自行接入第三方工具（邮件、日历、数据库查询等）
- 构建 "工具市场" 概念：用户可以安装、启用、配置社区工具
- 初期支持 3～5 个高频工具：邮件搜索、GitHub Issue 查询、本地文件搜索

**预期收益**：从 "知识库问答工具" 升级为 "个人 AI Agent 平台"，产生网络效应
**实施难度**：高（MCP 协议实现 + 工具管理 UI）

## 六、总结

项目当前处于 **"工程化 MVP"** 阶段——核心链路完整、代码质量高、已有评估体系，这是很好的基础。

下一步的核心矛盾是 **从 "能跑通" 到 "用得好"**：

- **短期优先**：查询改写 + 拒答机制 + embedding 加速，以最低成本获得最大体验提升
