# AGENTS.md

本仓库的工程标准是高压、克制、现代化。默认目标不是“能跑”，而是“正确、简洁、可维护、无赘余”。

## 核心原则

- 代码质量必须保持高标准。任何实现都要追求清晰的结构、稳定的语义、可验证的行为。
- 代码使用中文注释，切需要专业、规范，和代码高度相关。
- 每一行代码都必须有存在理由。不允许保留多余分支、多余抽象、多余参数、多余状态、多余文件。
- 代码要直达本质。优先选择最短、最清晰、最不绕的实现路径，拒绝表演式设计。
- 仓库要保持洁净。禁止脏提交、临时文件、失效脚本、无用依赖、注释掉的大段旧代码、遗留调试输出。
- 一切框架都需要参考最新的官方文档，使用context7 mcp

## 代码洁癖

- 禁止无意义封装。只有在确实降低复杂度、隔离边界或复用明显成立时才允许抽象。
- 禁止死代码。未使用的函数、变量、导入、配置、类型、常量必须删除。
- 禁止“先留着以后也许会用”。没有当前价值的代码不进入仓库。
- 禁止一次性兼容写法污染主路径。主流程必须保持单一、直接、可读。
- 命名必须准确、克制、专业。不要使用模糊词，也不要使用炫技式命名。

## 仓库洁癖

- 提交前必须清理调试痕迹，包括 `print`、临时日志、临时脚本、实验文件、草稿文件和本地导出物。
- 新增文件前先确认是否真的需要新文件。能在现有边界内解决的问题，不要扩散结构。
- 目录结构必须稳定、收敛。禁止把试验性内容混入正式模块。
- 文档必须与代码同步。过期文档、失效示例、错误说明与代码一样属于缺陷。

## 兼容性策略

- 禁止编写兼容性代码。
- 禁止为了旧运行时、旧平台、旧接口、旧框架、旧版本 SDK 保留分支、垫片、polyfill、fallback 或双轨实现。
- 禁止“新老两套都支持”的折中方案。选择一个当前有效、长期可维护的方案即可。
- 如果外部约束强制要求兼容旧环境，必须先明确说明约束来源、影响范围和移除条件，未经确认不得落地。

## 框架与依赖

- 禁止使用旧框架、过时 API、废弃写法和停止维护的依赖。
- 涉及框架、库、SDK、API 设计或升级时，必须优先使用 `context7 MCP` 查询当前稳定版本、官方文档和推荐范式，再进行实现或修改。
- 禁止凭记忆沿用旧版本经验。只要涉及现代框架，就以当前稳定版本文档为准。
- 新依赖必须有明确收益。能用标准库或现有依赖解决的问题，不新增依赖。

## 注释规范

- 注释必须整洁、专业、克制。
- 注释只解释代码表面看不出的信息：约束、不变量、边界条件、设计取舍、外部协议。
- 禁止把注释写成故事、教程、流水账或情绪表达。
- 禁止用注释复述代码字面含义。读代码就能知道的事情，不要再写一遍。
- 注释语言保持一致，术语准确，不使用模糊口语。

## 实现要求

- 优先消除复杂度，而不是转移复杂度。
- 优先收敛分支，而不是堆叠条件。
- 优先明确数据边界，而不是让状态在系统里漂移。
- 错误处理必须明确，不能吞错，不能制造静默失败。
- 测试或验证应覆盖真实行为边界，而不是只覆盖表面路径。

## 禁止事项

- 禁止提交未使用代码。
- 禁止提交注释掉的大段旧实现。
- 禁止提交“临时方案”但不标明替换计划。
- 禁止为猜测中的未来需求提前设计。
- 禁止为了“看起来完整”而增加无价值代码。

## 交付标准

- 改动后仓库必须比改动前更简洁，而不是更臃肿。
- 新代码必须融入现有结构，不制造新的混乱层。
- 如果一个实现不能用简洁、专业、可审查的方式解释清楚，说明它还不够好。
